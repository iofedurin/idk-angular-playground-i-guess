# Product Features — HR Portal

---

## Epic: Org Board — интерактивная доска организационной структуры

**Статус:** Идея
**Приоритет:** Следующий

---

### Концепция

Визуальная канвас-доска, на которой строится организационная структура компании. Слева — панель со списком всех сотрудников. Drag & drop переносит сотрудника на доску. На доске карточки можно свободно перемещать, соединять связями (manager → subordinate), группировать. По клику на карточку — детальная информация о сотруднике.

Это не просто "org chart viewer" — это **org chart builder**: HR буквально конструирует иерархию, перетаскивая людей и рисуя связи.

---

### Доска (Canvas area)

**Карточки сотрудников на доске:**
- Компактная карточка: аватар-инициалы, имя, должность, отдел
- Свободное позиционирование (absolute coordinates на доске)
- Drag для перемещения по доске
- Визуальное различие по роли (admin/editor/viewer — цветовой accent)
- Выделение по клику → sidebar с полной информацией о сотруднике

**Связи (edges):**
- Из каждой карточки можно вывести связь к другой карточке
- Связь = "этот сотрудник является менеджером для того" (`managerId`)
- У сотрудника может быть **один менеджер**, но **много подчинённых**
- Визуально: линия/кривая между карточками, стрелка указывает направление (manager → report)
- Связь можно удалить (drag от карточки или контекстное действие)

**Навигация по доске:**
- Pan (перетаскивание пустого пространства)
- Zoom in/out (колёсико мыши / pinch / кнопки)
- "Fit to screen" — автоматически подогнать масштаб чтобы все карточки были видны
- Minimap (опционально) — маленькая карта в углу для ориентации на больших досках

---

### Панель сотрудников (sidebar)

**Список:**
- Все сотрудники текущего workspace
- Поиск по имени
- Фильтр по отделу
- Визуальная индикация: кто уже на доске, кто ещё нет
- Drag из списка → drop на доску = добавить карточку

**Режим "информация":**
- По клику на карточку на доске sidebar переключается в режим детальной информации
- Показывает: полное имя, email, должность, отдел, страну, роль, bio
- Кнопки: "Перейти в профиль", "Редактировать"
- Список прямых подчинённых (кликабельные — фокусирует карточку на доске)
- Менеджер (кликабельный)

---

### Создание и управление связями

**Создание связи:**
- У каждой карточки — "connection point" (точка/якорь на краю карточки)
- Drag от connection point → визуальная линия следует за курсором → drop на другую карточку = создать связь
- Направление: drag **от менеджера к подчинённому** (интуитивно: "я назначаю тебе менеджера")
- При создании связи — PATCH `/api/users/:subordinateId { managerId: managerId }` → patchState → activity event

**Валидация связей:**
- **Нельзя создать цикл** — если A → B → C, то C не может стать менеджером A. Cycle detection при drop
- **Один менеджер** — если у сотрудника уже есть менеджер, новая связь **заменяет** предыдущую (с confirmation: "У Иванова уже есть менеджер Петров. Заменить на Сидорова?")
- **Нельзя быть менеджером самому себе** — drag на ту же карточку = no-op
- Visual feedback: при drag'е связи подсвечиваются валидные targets (зелёным) и невалидные (красным/серым)

**Удаление связи:**
- Клик по связи (линии) → выделение → Delete / кнопка "Удалить связь"
- Или через sidebar сотрудника: "Менеджер: Петров [×]"
- PATCH `/api/users/:id { managerId: null }` → карточка остаётся на доске, но без связи

### Каскадные эффекты и сложные сценарии

**Удаление сотрудника, у которого есть подчинённые:**
- Confirmation dialog: "У Петрова 5 прямых подчинённых. Что сделать с ними?"
  - Вариант 1: Переназначить на менеджера Петрова (если есть) — автоматический "подъём" на уровень выше
  - Вариант 2: Оставить без менеджера — подчинённые становятся "orphaned"
- Визуально на доске: связи перестраиваются анимированно

**Удаление карточки с доски (не удаление сотрудника!):**
- Карточка исчезает с доски, но сотрудник остаётся в системе
- Все связи этой карточки удаляются
- Сотрудник возвращается в sidebar-список как "не на доске"
- Менеджер/подчинённые связи в БД — что с ними? (интересный product question: доска = source of truth для связей, или отдельное представление?)

**Drag сотрудника между менеджерами (reassignment):**
- Перетащить карточку подчинённого на карточку другого менеджера = переназначить менеджера
- Альтернатива: пересоздать связь через connection points
- Activity event: "Иванов reassigned from Петров to Сидоров"

### Computed-данные и агрегации

**На карточке (опционально, по toggle):**
- Количество прямых подчинённых (badge на карточке)
- "Span of control" — все подчинённые рекурсивно (прямые + их подчинённые)

**Dashboard-агрегации (новые карточки на Dashboard):**
- Топ менеджеры по span of control
- Сотрудники без менеджера (orphaned)
- Максимальная глубина дерева
- Среднее количество прямых подчинённых

**На доске:**
- Auto-layout — кнопка "Упорядочить": алгоритм расставляет карточки по уровням иерархии (CEO сверху, уровни вниз). Сохраняет результат как позиции карточек
- Highlight subtree — по клику на менеджера подсветить всё его поддерево

---

### Persistence (что сохраняется)

- `User.managerId` — связь хранится на User entity (source of truth для иерархии)
- Позиции карточек на доске — отдельная коллекция `BoardPosition { userId, x, y }` (per-workspace)
- Какие сотрудники размещены на доске — derived из наличия BoardPosition
- Zoom / pan state — опционально, в localStorage или на backend

### Интересные моменты для реализации

| Тема | Почему интересно |
|---|---|
| **Self-referential User** | `managerId` ссылается на User — впервые entity ссылается сама на себя |
| **Cycle detection** | Нетривиальный алгоритм: при создании связи нужен BFS/DFS по дереву подчинённых |
| **Canvas-подобный UI без canvas** | DOM-элементы с absolute positioning, SVG-линии для связей, transform для zoom/pan |
| **Drag & drop 3 типа** | Sidebar → доска (добавление), карточка по доске (перемещение), connection point → карточка (связь) |
| **Optimistic UI** | Связь рисуется мгновенно, PATCH уходит async. При ошибке — rollback с анимацией |
| **Cross-entity cascade** | Удаление менеджера затрагивает подчинённых — UI должен показать preview изменений |
| **SVG edge rendering** | Линии между карточками: прямые? кривые Безье? Пересечения? Routing вокруг карточек? |
| **Auto-layout algorithm** | Hierarchical layout (Sugiyama или tree layout) — расставить N нод по уровням минимизируя пересечения |
| **Minimap** | Маленькая копия всей доски для навигации — масштабированный thumbnail в реальном времени |
| **Zoom/Pan** | CSS transform на контейнере, пересчёт координат мыши с учётом масштаба |
| **Board vs Entity state** | Позиции карточек — UI state (BoardPosition), связи — domain state (User.managerId). Два разных стора |
| **Large dataset** | 100+ сотрудников на доске — нужна виртуализация? Или DOM справится? |
| **Real-time collaboration** | WebSocket уже есть — можно показывать перемещения карточек другими пользователями в real-time |
| **Undo/redo** | Command pattern для действий на доске — особенно ценно при случайном удалении связи |

### Сценарии использования

**Сценарий 1: HR строит оргструктуру с нуля**
1. Открывает доску (пустая)
2. Перетаскивает CEO из sidebar'а на доску
3. Перетаскивает VP-ов, соединяет связями с CEO
4. Перетаскивает team leads, соединяет с VP-ами
5. Нажимает "Auto-layout" → доска красиво раскладывается по уровням
6. Сохраняет (позиции + связи персистятся)

**Сценарий 2: Реорганизация отдела**
1. Открывает доску с существующей структурой
2. Видит что у менеджера X слишком много подчинённых (badge "12")
3. Создаёт нового team lead (переход на страницу создания → обратно на доску)
4. Перетаскивает 6 подчинённых от X к новому team lead
5. Каждое перетаскивание = PATCH + activity event
6. В audit log: 6 записей "reassigned from X to Y"

**Сценарий 3: Просмотр "кто кому подчиняется"**
1. Клик по менеджеру → подсветка всего поддерева
2. Sidebar показывает: 3 прямых подчинённых, 12 всего в поддереве
3. Клик по подчинённому в sidebar → камера центрируется на его карточке
4. Zoom out → видна вся организация

**Сценарий 4: Увольнение менеджера**
1. Клик на карточку менеджера → sidebar → "Удалить сотрудника"
2. Dialog: "У Петрова 5 подчинённых. Переназначить на его менеджера (Директор)?"
3. OK → 5 связей перестраиваются анимированно → карточка исчезает с доски
4. Activity: "Петров deleted", "Иванов reassigned to Директор" × 5
